# 0002 — Capabilities manifest for skills & MCP servers

- **Status:** Accepted
- **Date:** 2026-07-06

## Context

Scaffolded projects can accumulate skills (`.agents/skills/`) and MCP servers
(`.mcp.json` / `.cursor/mcp.json`) via `neptr skill`, `neptr mcp`, raw
`npx skills add`, and the feature/adopt implement phases. Agents were told only to
"skim `.agents/skills/`" and were left to read raw MCP JSON config, with no curated
statement of what each capability is for or which to prefer. That invites two failure
modes: **over-selection** (using too many tools) and **conflicts** (overlapping or
contradictory capabilities).

A plain hand-maintained "skills manifest + MCP manifest" was considered, but any file
listing installed capabilities would drift the moment any of the above install paths
runs without updating it — exactly the staleness `neptr index` already exists to
prevent.

## Decision

We add a single per-project **`.agents/CAPABILITIES.md`** with two zones:

1. **Auto-generated inventory** — a Skills table and an MCP servers table between
   `<!-- neptr:skills:start/end -->` and `<!-- neptr:mcp:start/end -->` markers,
   generated by `neptr index` (scanning `.agents/skills/*/SKILL.md` frontmatter and
   parsing `.mcp.json`). Byte-stable and kept fresh by the existing SessionStart and
   pre-commit hooks, so it cannot drift.
2. **Hand-written usage policy** (prose outside the markers) — "a capability must be
   listed here to be used", "prefer the fewest tools", "one capability, one tool" with
   a human-curated precedence list, and "don't install what's already covered". This is
   the part that actually curbs over-use and conflicts.

The file is wired into required reading: the generated root agent files point at it
before any skill/MCP use, `AI_INSTRUCTIONS.md` and the skills README reference it, and
the feature/adopt implement phases record overlaps in its precedence list.

We deliberately kept v1 to inventory + policy — no automated overlap-detection
heuristic — to keep the generator simple and deterministic.

## Consequences

- Agents get one always-read, never-stale place describing installed capabilities and
  how to choose among them, reducing tool sprawl and conflicts.
- `neptr index` (and its hooks) now also owns `.agents/CAPABILITIES.md`; the
  pre-commit hook stages it alongside `REPO_MAP.md` and `KNOWLEDGE_MAP.md`.
- Projects scaffolded before this feature have no `CAPABILITIES.md`; `neptr index`
  skips the refresh gracefully until one exists (added by re-scaffolding or by hand).
- The manifest lists capabilities but does not enforce them at runtime — enforcement
  is the agent following the policy, not tooling.
